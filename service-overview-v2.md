# Service Overview v2 (개선안)

> 본 문서는 기존 `service-overview.md`(v1)의 후속으로, 1개월간 운영하면서 발견한 "데이팅앱 톤" 문제를 해결하기 위한 개선 방향을 정의합니다.
>
> v1 대비 변경 사항 위주이며, 변경되지 않은 영역(PWA 구조, Supabase Auth, 디자인 시스템 등)은 기존 `TECHNICAL-SPEC.md` 참조.
>
> **작성일**: 2026-05-05

---

## 1. 컨셉 재정의

### 1.1 v1의 문제 진단

지금까지 만든 구조는 "초대 코드"라는 진입 장벽만 있을 뿐, 그 안의 동선은 데이팅앱과 거의 동일합니다.

- 참가자가 추천 카드를 보고 자율적으로 찜 → 상호 찜 → 자동 매칭
- 인기도/외모 점수가 노출 우선순위에 영향 (`boost_until`, `look_score`, `popularity factor`)
- 프로필 방문자 추적 (`profile_views`)
- 단일 점수(95점 등)로 매칭 적합도 노출

이는 "결정사·데이팅앱은 부담스럽고, 지인이 연결해주는 소개팅"이라는 본래 의도와 정면으로 충돌합니다.

### 1.2 v2의 핵심 명제

**"매칭은 사람(주선자)이 한다. 시스템은 주선자의 도구일 뿐이다."**

- 참가자는 매칭 의사결정을 하지 않음. **소개를 받고 yes/no만 한다.**
- 알고리즘은 매칭을 "결정"하지 않음. **주선자가 두 사람의 궁합을 추측하는 보조 도구.**
- 풀(pool)은 시스템이 정하지 않음. **주선자가 자기 인간관계 안에서 풀을 만든다.**

## 2. 사용자 역할

| 역할 | 설명 | 진입 |
|------|------|------|
| 참가자 (participant) | 소개를 받아 만남을 갖는 대상자 | 주선자 또는 참가자의 추천 코드로 가입 |
| 주선자 (matchmaker) | 지인을 소개하는 사람 | 관리자 슈퍼코드 또는 추천 코드로 가입 후 자기 코드 배포 |
| 관리자 (admin) | 시스템 운영 | `admin_users` 테이블 |

**v1 대비 변경: 한 사람이 참가자와 주선자를 동시에 겸할 수 있음.** 본인이 소개를 받기도 하고, 친구를 추천하기도 하는 것이 자연스러운 형태이기 때문.

- 스키마: `applicants.role`(단일 컬럼) → `is_participant BOOLEAN`, `is_matchmaker BOOLEAN` 두 컬럼
- UI: 별도의 `matchmaker.html`을 폐기하고, `dashboard.html`에 "주선자 모드" 탭/토글 추가

## 3. 풀(pool) 정책 -- C안 채택

> 한 주선자가 어떤 참가자에게 접근할 수 있는가의 문제. 컨셉의 90%가 여기서 결정됨.

### 3.1 기본 풀

각 주선자의 기본 풀 = **본인이 추천 코드로 직접 초대한 사람들(1촌)만**.

이 외의 모든 참가자 정보는 기본적으로 비공개. 다른 주선자의 풀에 누가 있는지조차 보이지 않음.

### 3.2 풀 확장 = 주선자 간 협업

자기 풀에 적합한 후보가 없을 때, 주선자는 다른 주선자에게 요청을 보낼 수 있음. 두 가지 방식 모두 허용:

**(a) 조건 broadcast** — 모든 주선자에게 조건만 적은 요청 발송
- 예: "30대 초반, 서울 거주, 전문직, 유머감각 있는 분"
- 받은 주선자가 자기 풀에 해당하는 사람이 있을 때만 응답
- 제약: 한 요청당 최대 3건의 응답까지 받음, 7일 후 자동 만료

**(b) 직접 지목** — 친한 다른 주선자 1명에게 1대1 요청
- "혹시 우리 친구 X한테 어울릴 분 있어?"

응답한 주선자(B)가 자기 풀의 Y를 추천하기로 동의하면, 그제서야 요청자(A)에게 Y의 프로필이 한정 공개됨.

## 4. 소개 프로세스 (전체 흐름)

```
[1] 주선자 A가 자기 사람 X에게 누구 소개할지 고민
        │
        ▼
[2] A의 1촌 풀에서 후보 검색
        │  - 호환성 리포트(섹션 5)가 표시됨
        │  - 적합한 후보 있음 → [5]로 점프
        ▼
[3] 풀에 후보 없으면 다른 주선자에게 요청
        │  (a) 조건 broadcast (모든 주선자에게)
        │  (b) 직접 지목 (특정 주선자 1명에게)
        ▼
[4] 다른 주선자 B가 "우리 Y 어때?" 응답
        │  - B의 동의 후 Y의 프로필 + 평판이 A에게 한정 공개
        │  - A가 검토 후 "별로" → 종료, B에게 통보
        │  - "좋다" → 다음 단계
        ▼
[5] X와 Y에게 정식 소개 알림 발송
        │  - 소개자 표시: "A·B 공동 소개" (B 없는 경우 A 단독)
        │  - X와 Y 양쪽이 서로의 프로필을 봄 (이때 처음으로 양쪽에 공개)
        ▼
[6] X, Y 각자 yes/no
        - 양쪽 yes → 자동 매칭 → 채팅 개설
        - 한쪽 no → 종료, 양쪽 주선자에게 결과 통보
```

### 4.1 동시 다중 소개 정책

- **한 참가자에게 여러 주선자가 동시에 소개를 보낼 수 있다.** (제약 없음)
- **참가자도 동시 여러 소개에 yes를 누를 수 있다.**
- **여러 매칭이 동시에 성사되어도 모두 유지된다.** 한 사람이 동시에 여러 명과 매칭 상태로 채팅 가능.
  - v1의 단일 매칭(`matched_with` 단일 FK)은 더 이상 적합하지 않음 → 별도 `matches` 테이블에서 다대다 관계로 변경.
  - 가입 시 사용자에게 "이 서비스는 동시 여러 만남을 허용합니다"를 명시적으로 안내.

### 4.2 매칭 종료/해제

- 두 사람 중 한 명이 명시적으로 "관계 종료"를 누르면 그 매칭은 종료
- 양쪽 30일 이상 미접속 또는 메시지 없으면 자동 비활성화
- 종료된 매칭의 채팅 이력은 유지 (조회만 가능, 새 메시지 발송 불가)

## 5. 매칭 점수 알고리즘

### 5.1 표시 방식 변경: 점수 → 호환성 리포트

v1은 단일 점수(95점)를 참가자에게 노출했음. v2는 **주선자에게만**, **점수가 아닌 항목별 요약**으로 표시.

```
[X님 입장에서 본 Y님]
- 키, 직업: 이상형 범위 안 ✓
- MBTI: 보통 (ENFP × ISFJ)
- 지역: 거리 있음 (서울 ↔ 분당)
- 종교: 다름

[Y님 입장에서 본 X님]
- 나이, MBTI: 이상형 일치 ✓
- 직업: 이상형 명시 안 함
- 지역: 동일 (서울)
```

주선자가 정성적으로 판단할 수 있도록 정보를 **문장 형태**로 제공. 단일 점수는 어디에도 노출하지 않음.

### 5.2 알고리즘 자체

- v1의 `calcMatchScore` 내부 로직(가중치, MBTI 매트릭스, 직업 점수)은 유지
- 출력 단계만 변환: 점수 → 항목별 매치 여부 텍스트
- **`look_score` 항목은 알고리즘에서 제거** (관리자 외모 점수 부여 폐기)

## 6. 평판 시스템

### 6.1 트리거 재정의

v1: 임의 시점에 평판 작성
v2: **A가 B의 추천 코드로 가입했으면, B가 A에 대해 평판을 작성하는 것이 의무**

### 6.2 필수 게이트 흐름

```
1. A가 B의 추천 코드로 가입 신청
2. A.status = pending_reputation
3. B에게 "A님에 대한 평판 작성 부탁드립니다" 알림 + 푸시
4. B가 평판 작성 완료
5. A.status = pending (관리자 검토 단계)
6. 관리자 검토 → A.status = approved
```

평판이 없으면 A는 가입 절차를 완료할 수 없음. 즉 **평판 = 가입의 필수 게이트**.

### 6.3 안전장치 (B가 평판을 안 써주는 경우)

평판 의존성이 사용자를 영구히 막아버리는 위험 완화:

- 가입 1일째: B에게 부드러운 reminder 푸시
- 3일째: 두 번째 reminder
- 7일째: 관리자 에스컬레이션 큐로 자동 이동
  - 관리자가 직접 A를 인터뷰
  - 또는 A가 다른 지인의 추천 코드로 재가입하도록 안내

### 6.4 평판 공개 범위

| 대상 | 공개 여부 |
|------|-----------|
| 작성자 본인(B) | 자기가 쓴 평판 열람 가능 |
| 평판 대상자(A) | 자기에 대한 평판 열람 가능 (작성자 익명 여부는 추후 결정) |
| A의 추천인(=B 본인) | 위와 동일 |
| 매칭 상대 측 주선자 | A가 다른 주선자 C의 풀에 소개될 때, 소개 흐름 [4]단계에서 C에게 한정 공개 |
| 매칭 상대 본인 | 비공개 (주선자가 검토 후 정성적으로 전달) |

## 8. 화면 구조 변화

### 8.1 참가자 화면 (`dashboard.html`)

| v1 | v2 |
|----|----|
| 추천 / 관심 / 대화 / MY (4탭) | **소개 / 대화 / MY (3탭)** |

- **소개 탭**: 받은 소개 목록 (yes/no 대기 중, 응답 완료, 만료된 소개 분류)
- **대화 탭**: 매칭된 사람들과의 채팅. 다대다 매칭이므로 여러 채팅방 동시 존재 가능
- **MY 탭**: 프로필 관리, 푸시 설정, 내 평판 보기, 문의, 로그아웃

### 8.2 주선자 화면

`matchmaker.html`을 폐기하고 `dashboard.html`에 통합. 1인 2역 사용자는 화면 상단 토글로 모드 전환.

| v1 (`matchmaker.html`) | v2 (`dashboard.html` 주선자 모드) |
|----|----|
| 추천 현황 / 소개 / 현황 (3탭) | **내 사람들 / 소개하기 / 요청함 / 이력 (4탭)** |

- **내 사람들**: 내 추천 코드로 가입한 사람 목록, 상태 / 평판 / 매칭 현황
- **소개하기**: 내 사람 X 선택 → 자기 풀에서 후보 검색 또는 다른 주선자에게 요청 (broadcast / 직접 지목)
- **요청함**: 다른 주선자가 보낸 broadcast 요청 목록, 내 풀에서 매칭되는 사람 추천
- **이력**: 내가 진행한 소개의 상태 (pending / matched / declined / expired)

## 9. 빼는 것 / 남기는 것 / 새로 만드는 것 요약

### 빼는 것 (데이팅앱 색을 강화하던 요소)
- 추천 카드 / 찜 / 상호 찜 / 자동 매칭
- 인기도, 외모 점수, 부스트, 프로필 방문자
- 워터마크 (스크린샷 방지)
- 매칭 점수의 단일 숫자 노출
- 일일 소개 한도 (matchmaker tier)

### 남기는 것 (서비스 정체성)
- 초대 코드 가입
- 평판 시스템 (트리거 재정의 + 필수 게이트)
- 매칭 점수 알고리즘 (주선자에게만 + 요약 형태)
- 채팅 (매칭 후에만)
- 관리자 검토

### 새로 만드는 것 (C안의 핵심)
- 주선자 → 주선자 요청 시스템 (broadcast + 직접 지목)
- 다대다 매칭 (한 명이 여러 명과 동시 매칭 가능)
- 1인 2역 (참가자 + 주선자 동시 활동)
- 주선자용 호환성 리포트 UI

## 10. 정보 보호 정책

### 10.1 배경

지인 소개 컨셉은 데이팅앱보다 캡쳐 사고에 훨씬 취약함. 실명·실제 사진·실제 신상이 친구·지인 네트워크 안에서 도는 구조이기 때문. 특히 v2에서 새로 추가되는 화면(broadcast 요청 본문, 호환성 리포트, 평판, 소개 상세 등)은 모두 캡쳐 시 당사자에게 부담이 큰 정보를 담음.

PWA 환경에서는 OS 레벨 스크린샷 차단이 불가능하므로, **"캡쳐 자체를 하지 않게 만드는 deterrent"** 와 **약관에 의한 사후 제재**의 두 축으로 보호함.

### 10.2 워터마크 정책

- **모든 화면에 균일하게 적용** (강도 차등화 없음 - 코드 복잡도 회피)
- **워터마크 내용**: `{본인 이름} · {본인 전화번호} · {YYYY-MM-DD HH:mm}`
- **목적**: 캡쳐 시 본인의 개인정보가 같이 유출되도록 하여 캡쳐 자체를 억제
- **구현**: v1의 `applyWatermark()` 유지 (Canvas 타일 패턴, 회전 -25°, rgba 0.07, position:fixed, pointer-events:none)
- **부작용 인지**: 캡쳐 사고 시 캡쳐자 본인의 전화번호가 함께 유출됨. 이는 의도된 압박 효과이며, 가입 시점에 약관으로 명시 동의 받음.

### 10.3 약관/정책 명시 사항

가입 시 명시 동의를 받아야 하는 항목:

1. **스크린샷 외부 공유 금지**: 본 서비스의 모든 화면(프로필, 평판, 호환성 리포트, broadcast 요청, 소개 알림, 채팅 등)을 캡쳐하여 외부에 공유하는 것을 금지함. 위반 시 영구 강제 탈퇴 및 법적 책임을 질 수 있음.
2. **워터마크 동의**: 모든 화면에 본인의 이름, 전화번호, 접속 일시가 워터마크로 표시되며, 캡쳐 시 함께 유출됨에 동의함.
3. **다대다 매칭 안내**: 본인이 동시에 여러 명과 매칭 상태일 수 있으며, 상대방도 동일함을 인지함.
4. **평판 시스템 안내**: 본인이 추천한 사람에 대해 평판을 작성할 의무가 있으며, 작성된 평판은 추후 매칭 과정에서 다른 주선자에게 공개될 수 있음을 인지함.

### 10.4 위반 발견 시 절차

- 외부 채널(SNS, 단톡방 캡쳐 등)에서 본 서비스의 캡쳐가 발견되면, 워터마크의 이름·전화번호로 캡쳐자 식별
- 1차 위반: 경고 + 30일 정지
- 2차 위반: 영구 강제 탈퇴
- 악의적 유포가 명백한 경우: 1차부터 영구 강제 탈퇴 + 법적 조치 검토

## 11. 마이그레이션 단계 제안

기존 코드와 데이터를 v2로 점진적 이전:

**Phase 1 — 스키마 마이그레이션**
- 신규 컬럼/테이블 추가 (호환 유지)
- 기존 `role` → `is_participant`/`is_matchmaker` 백필

**Phase 2 — 동선 차단**
- 추천 카드 / 찜 UI 비활성화 (코드는 유지, 점진 폐기)
- 자율 매칭 RPC 호출 차단

**Phase 3 — 신규 동선 구현**
- 주선자 간 요청 시스템
- 호환성 리포트 UI
- 다대다 매칭

**Phase 4 — 데드 코드 정리**
- 폐기 테이블 / 컬럼 / RPC / 화면 삭제

## 12. 추후 결정 / 검증 필요

| 항목 | 메모 |
|------|------|
| 평판 작성자 익명 vs 실명 | A가 자기 평판을 볼 때 누가 썼는지 보이는가? |
| 주선자 자격 제한 | 모든 가입자가 자동으로 주선자 활동 가능한가, 아니면 별도 자격(예: 가입 후 30일, 본인 평판 ≥1건)이 필요한가? |
| broadcast 노이즈 관리 | 서비스 성장 시 모든 주선자에게 broadcast가 가면 시끄러움. 지역/연령 사전 필터링 필요 가능. |
| 다대다 매칭 검증 | "한 사람이 여러 명과 동시 매칭"이 컨셉(가벼운 지인 소개)과 약간 결이 다름. 사용자 반응 본 뒤 단일 매칭으로 회귀 여부 재검토. |
| 평판 필수 게이트의 사용자 이탈 | 가입 → 평판 대기 → 7일 → 승인까지 마찰이 큼. 가입 시점에 명확한 안내 필요. |
| 1인 2역 UX | 모드 전환 토글이 직관적인지, 아니면 두 화면을 한 페이지에 통합하는 게 나은지 사용자 테스트 필요. |
